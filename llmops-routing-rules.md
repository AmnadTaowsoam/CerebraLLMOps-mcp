# LLMOps Routing Rules

**Standard ID:** STD-LLMOPS-ROUTING-001
**Category:** llmops
**Priority:** critical
**Applies To:** all-tasks
**Owner MCP:** CerebraLLMOps-mcp

## 1. Purpose

This document defines when and how callers (Orchestrator and downstream MCPs) must interact with CerebraLLMOps-mcp. It specifies which tool to call, under which conditions, and what to do with the result.

---

## 2. When to Call `llmops_get_model_routing_decision`

### Mandatory Triggers

Call `llmops_get_model_routing_decision` **BEFORE initiating any LLM call** when any of the following conditions are true:

| Condition | Rule |
|-----------|------|
| `task_risk = medium` | REQUIRED — default model may not match risk level |
| `task_risk = high` | REQUIRED — high-risk tasks require validated model selection |
| `task_risk = critical` | REQUIRED — always route through LLMOps; override not allowed |
| `multi_mcp = true` | REQUIRED — cross-MCP tasks need coordinated model selection |
| `task_type = code_generation` | REQUIRED — code generation tasks have special model requirements |
| `task_type = debug` | REQUIRED — debug tasks need high-accuracy models |
| `task_type = review` | REQUIRED — review tasks need models with strong reasoning |
| `task_type = security_analysis` | REQUIRED — security tasks must use approved models only |
| `task_type = deployment` | REQUIRED — deployment decisions require critical-level model |
| `tenant_budget_constraint = true` | REQUIRED — respect per-tenant token budget limits |

### Optional (Recommended)

| Condition | Rule |
|-----------|------|
| `task_risk = low` and `task_type` is simple | MAY use default model — but LLMOps routing is always preferred |
| Single-MCP read-only tasks | MAY skip if the task has no LLM component |

### Inputs Required

```typescript
{
  task_type: string,       // from skill or task declaration
  risk_level: "low" | "medium" | "high" | "critical",
  tenant_id: string,
  context_size_estimate?: number,   // tokens estimated before call
  cost_budget_override?: number,    // per-call override (platform owner only)
}
```

### What You Get Back

```typescript
{
  model_id: string,          // e.g. "claude-sonnet-4-5"
  fallback_model_id: string, // e.g. "claude-haiku-3-5"
  token_budget: number,      // max input tokens for this call
  eval_requirement: "none" | "automated" | "automated+human",
  guardrail_level: "standard" | "strict" | "maximum",
  human_review_required: boolean,
  routing_reason: string,    // explanation for audit trail
}
```

## 3. When to Call `llmops_get_context_package`

Call this when you need the full `LLMOpsContextPackage` (see `llmops-context-package-schema.md`).

The Orchestrator calls this at the start of every managed run. Downstream MCPs should NOT call this directly — they receive context from the Orchestrator's composed package.

### Timing: Before the LLM call, after `orchestrator_compose_learning_recommend`

```
Orchestrator flow:
  1. orchestrator_compose_learning_recommend(context)
  2. llmops_get_context_package(task, risk, tenant)    ← here
  3. Initiate LLM call with model_id + token_budget
  4. llmops_record_llm_trace(trace_data)              ← after
```

---

## 4. When to Call `llmops_record_llm_trace`

**REQUIRED** after every managed LLM run. No exceptions.

### Timing: Immediately after the LLM response is received

### Required Fields

```typescript
{
  trace_id: string,          // UUID generated by caller
  agent_id: string,
  task_id: string,
  prompt_version: string,    // semver of prompt used
  model_id: string,          // actual model used (may differ from routing if fallback triggered)
  input_tokens: number,
  output_tokens: number,
  cost_estimate: number,     // USD, calculated from model-cost-latency-profile
  latency_ms: number,
  tool_calls: string[],      // list of tools called during run
  eval_result: "pass" | "fail" | "skip" | "pending",
  status: "success" | "failure" | "partial",
}
```

### Do NOT Skip Recording When

- The run failed (failure traces are required for failure pattern analysis)
- The run used a fallback model (record which model was actually used)
- The run was partial (partial traces help diagnose truncation issues)

## 5. When to Call `llmops_record_feedback`

Call `llmops_record_feedback` when a downstream MCP produces a finding about an LLM output.

### Mandatory Triggers

| Trigger Source | When to Call |
|----------------|-------------|
| **CerebraReview-mcp** | After every review finding — pass or fail |
| **CerebraDebugs-mcp** | After a diagnostic run produces a root cause or suggested fix |
| **CerebraTesting-mcp** | After a quality gate evaluation |
| **CerebraSecurity-mcp** | After any security policy violation or DLP trigger |
| **CerebraStandards-mcp** | After a standards compliance check with findings |
| **Human reviewer** | After a human reviews and annotates an agent output |

### Feedback Severity Mapping

| Finding | Severity |
|---------|---------|
| Critical security violation | `critical` |
| Eval failure (automated) | `high` |
| Standards non-compliance | `medium` |
| Minor style/format issue | `low` |
| Positive finding (good output) | `info` |

## 6. When to Call `llmops_evaluate_output`

Call this when you need an explicit quality evaluation on a completed output.

### Mandatory Cases

| Case | Evaluation Type |
|------|----------------|
| `risk_level = critical` | `automated + human` — both required |
| `risk_level = high` | `automated` — human review if automated fails |
| New prompt version (first run) | `automated + regression` |
| Deployment-type task | `automated` always |
| Output claimed as evidence (ticket closure) | `automated` always |

### Optional Cases

| Case | Recommendation |
|------|---------------|
| `risk_level = medium` | Run automated eval if latency budget allows |
| `risk_level = low` | Skip unless the task type has a mandatory eval binding |

---

## 7. Escalation Triggers

The following conditions MUST cause immediate escalation to human review. Escalation is initiated via `llmops_record_feedback` with `severity=critical` and `requires_human_review=true`.

| Trigger | Escalation Action |
|---------|------------------|
| `eval_result = fail` and `risk_level = critical` | Block output — do not deliver to user |
| Guardrail triggered (output contains PII/secrets) | Strip output, flag for review |
| Model returned confidence < 0.5 (self-reported) | Deliver with disclaimer, flag for review |
| Hallucination detected (citation not found in source) | Block output, require re-run with grounding |
| Tool call to a tool outside agent's permission matrix | Deny tool call, log violation, escalate |
| `cost_estimate` exceeds tenant's per-run budget | Block run, notify tenant owner |
| Latency > 30s for any single LLM call | Log latency violation, alert on-call |
| Agent attempts to modify its own instructions | Block and escalate immediately |

## 8. Decision Flowchart

```
Agent starts a task
       |
       v
Is there an LLM call involved?
  NO → skip LLMOps routing (proceed directly)
  YES ↓
       |
       v
Call llmops_get_model_routing_decision
  → Returns: model_id, token_budget, guardrail_level, eval_requirement
       |
       v
Is risk_level = critical OR multi_mcp = true?
  YES → Also call llmops_get_context_package for full package
  NO  → Proceed with routing decision only
       |
       v
Execute LLM call with model_id + token_budget
       |
       v
[After response received]
Call llmops_record_llm_trace (ALWAYS)
       |
       v
Is eval_requirement = "automated" or "automated+human"?
  YES → Call llmops_evaluate_output
  NO  → Skip eval
       |
       v
Did downstream MCP produce findings?
  YES → Call llmops_record_feedback
  NO  → Done
```
