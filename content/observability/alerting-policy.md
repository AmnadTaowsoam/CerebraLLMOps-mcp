# Alerting Policy

**Standard ID:** STD-LLMOPS-ALERTING-001
**Category:** llmops/observability
**Priority:** high
**Owner MCP:** CerebraLLMOps-mcp

## 1. Purpose

Defines alert conditions, severity levels, routing rules, and response SLAs for CerebraLLMOps observability alerts.

---

## 2. Alert Severity Levels

| Severity | Response SLA | Who Gets Notified | Escalation |
|---------|-------------|------------------|-----------|
| `critical` | 15 minutes | On-call + Platform owner + nottoei | Immediate |
| `high` | 1 hour | On-call + Platform owner | After 1h if unacknowledged |
| `medium` | 4 hours | Platform owner | After 4h if unacknowledged |
| `low` | Next business day | Dashboard + weekly report | No auto-escalation |

---

## 3. Alert Conditions

### Cost Alerts

| Alert ID | Condition | Severity | Notification |
|---------|-----------|---------|-------------|
| `COST-001` | Daily spend > 2× previous-day spend | high | On-call |
| `COST-002` | Daily spend > 80% of daily budget | medium | Platform owner |
| `COST-003` | Daily spend reaches 100% of daily budget | high | Platform owner + block |
| `COST-004` | Single run `cost_estimate > $5.00` | medium | Log + flag |
| `COST-005` | Monthly spend > 90% of monthly budget | critical | nottoei + platform owner |

### Latency Alerts

| Alert ID | Condition | Severity | Notification |
|---------|-----------|---------|-------------|
| `LAT-001` | Any single run exceeds Max Allowed latency | high | On-call |
| `LAT-002` | p95 latency exceeds SLO for > 10 runs / hour | critical | On-call + platform owner |
| `LAT-003` | Model timeout (> 60s) on any call | high | On-call |
| `LAT-004` | Context assembly > 5,000ms per run | low | Dashboard |
| `LAT-005` | Agent p95 latency > 3× baseline (7-day avg) | medium | Platform owner |

### Quality Alerts

| Alert ID | Condition | Severity | Notification |
|---------|-----------|---------|-------------|
| `QUAL-001` | Agent eval pass rate drops below 80% | high | Platform owner |
| `QUAL-002` | Eval failure rate > 20% for any agent in a day | critical | nottoei + platform owner |
| `QUAL-003` | Hallucination detection rate > 5% | high | Platform owner |
| `QUAL-004` | New eval case failure not previously failing | medium | Platform owner |
| `QUAL-005` | Human review SLA breach rate > 10% | medium | Platform owner |

### Safety Alerts

| Alert ID | Condition | Severity | Notification |
|---------|-----------|---------|-------------|
| `SAFE-001` | Credential detected in context (any occurrence) | critical | nottoei + on-call immediately |
| `SAFE-002` | PII detected in LLM output (not scrubbed) | critical | nottoei + on-call immediately |
| `SAFE-003` | Forbidden tool call attempted | high | On-call |
| `SAFE-004` | Multi-tenant data isolation violation | critical | nottoei + on-call immediately |
| `SAFE-005` | Guardrail trigger rate > 1% for any guardrail | medium | Platform owner |

### Model Reliability Alerts

| Alert ID | Condition | Severity | Notification |
|---------|-----------|---------|-------------|
| `MODEL-001` | Model API error rate > 1% | medium | On-call |
| `MODEL-002` | Model fallback rate > 5% | high | On-call |
| `MODEL-003` | Circuit breaker opens for any model | critical | On-call + platform owner |
| `MODEL-004` | Primary model (opus-4) unavailable for critical tasks | critical | nottoei + on-call |

---

## 4. Alert Routing

All alerts are routed through the standard Cerebra alerting pipeline:

1. Alert generated by LLMOps service
2. Published to `cerebra.alerts` topic
3. CerebraDevOps-mcp alert handler picks up and routes based on severity
4. `critical`/`high`: PagerDuty + Slack `#cerebra-alerts`
5. `medium`: Slack `#cerebra-platform` + email
6. `low`: Dashboard only

---

## 5. Alert Suppression

To prevent alert storms:
- Same alert ID is suppressed if it fires > 3 times in 10 minutes (deduplicated)
- During planned maintenance windows: all `medium` and `low` alerts are suppressed
- `critical` alerts are NEVER suppressed

---

## 6. Alert Response Runbook (Summary)

| Alert | First Response |
|-------|--------------|
| `COST-003` (budget reached) | Check cost dashboard, identify high-cost agent/task, consider downgrading model |
| `LAT-002` (sustained latency) | Check circuit breaker status, model API health, consider fallback activation |
| `QUAL-002` (high eval failure) | Pull failed cases, check for common pattern, check for prompt/model regression |
| `SAFE-001` (credential in context) | Identify source of credential, remove from knowledge base, rotate credential |
| `MODEL-003` (circuit breaker) | Monitor circuit breaker status, check Anthropic API status page |
